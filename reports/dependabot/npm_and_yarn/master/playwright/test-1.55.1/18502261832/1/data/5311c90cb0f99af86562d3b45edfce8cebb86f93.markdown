                - heading "Trusted Boot Keys" [level=1] [ref=e353]
                - paragraph [ref=e354]: All security features of Trusted Boot rely on cryptographic keys. Secure Boot relies on the chain of trust between Platform Key (PK), Key Exchange Keys (KEK), and Signature Database (DB) keys to achieve a tamper proof boot process. Full Disk Encryption also relies on a key pair. This section talks about the different keys used in Trusted Boot, their roles, and best practices to manage them securely.
                - paragraph [ref=e355]:
                  - img "Relationship between Secure Boot Keys" [ref=e356]
                - heading "Platform Key (PK)Direct link to Platform Key (PK)" [level=2] [ref=e357]:
                  - text: Platform Key (PK)
                  - link "Direct link to Platform Key (PK)" [ref=e358] [cursor=pointer]:
                    - /url: "#platform-key-pk"
                    - text: "#"
                - paragraph [ref=e359]: The PK is at the top of the Secure Boot cryptographic key hierarchy. It's a key pair that has a private key and a public key.
                - paragraph [ref=e360]: The private PK signs updates to the Key Exchange Key (KEK) variable, which contains a list of certificates, public keys, or signatures in a EFI Signature List (ESL). The public PK is used to verify whether updates to the KEK are signed with the authentic private key and can be trusted. It establishes a relationship of trust between the platform owner and the platform firmware. A system can only have one PK.
                - heading "Key Exchange Key (KEK)Direct link to Key Exchange Key (KEK)" [level=2] [ref=e361]:
                  - text: Key Exchange Key (KEK)
                  - link "Direct link to Key Exchange Key (KEK)" [ref=e362] [cursor=pointer]:
                    - /url: "#key-exchange-key-kek"
                    - text: "#"
                - paragraph [ref=e363]: The KEK is a set of certificates, public keys, or signatures in an ESL signed by a private PK key. Entries in the KEK set are used to update the Signature Database (DB) and the Forbidden Signature Database (DBX). Each entry in the KEK set has a corresponding private key. It establishes a relationship of trust between the firmware and the operating system (OS).
                - paragraph [ref=e364]: Any private key corresponding to an entry in the KEK set can sign updates to the DB. When there are updates to the DB, the corresponding public key, certificate, or signature in the KEK set is used to verify that those updates are signed by the authentic private key.
                - heading "Signature Database (DB) and Forbidden Signature Database (DBX)Direct link to Signature Database (DB) and Forbidden Signature Database (DBX)" [level=2] [ref=e365]:

                - paragraph [ref=e367]: "The DB validates signed EFI (Extensible Firmware Interface) binaries. The DB may contain a mixed set of certificates, public keys, signatures or hashes of binary files. The signature stored in the EFI binary (or a hash of the binary if there is no signature) is compared against the entries in the database. The binary will be executed if one of the following conditions is met:"
                - list [ref=e368]:
                  - listitem [ref=e369]: The EFI binary is signed, and the signing key is in the DB.
                  - listitem [ref=e370]: The EFI binary is signed, and the signature on the binary is in the DB.
                  - listitem [ref=e371]: The EFI binary is unsigned and a SHA-256 hash of the image is in the DB.
                - paragraph [ref=e372]: Similarly, DBX may contain a mixed set of certificates, signatures or hashes. Any EFI whose signatures, hashes, or signing key matches the entries in DBX is forbidden from being executed.

                - paragraph [ref=e376]:
                  - text: During EdgeForge, each boot component is hashed and these hash values, or measurements, are signed by the PCR private key. The signed measurements are embedded in the UKI image, along with the public key of the PCR policy key pair. For more information about EdgeForge, refer to
                  - link "EdgeForge with Trusted Boot" [ref=e377] [cursor=pointer]:

                - paragraph [ref=e378]: During installation, encrypted partitions are setup using a Disk Encryption Key (DEK), which is itself encrypted by the TPM and stored in a secure blob. The PCR public key embedded in the ISO is used to form a binding policy. The binding policy states that in order to decrypt the secure blob containing the DEK, the PCR measurements must match a pre-calculated set of measurements signed by the corresponding PCR private key.
                - paragraph [ref=e379]: "During the boot process, before the encrypted disk partition is mounted, the TPM will perform the following:"
                - list [ref=e380]:

                - paragraph [ref=e384]: If all three verifications are successful, TPM will decrypt the secure blob, release DEK, and the OS can use it to decrypt the encrypted partitions of the disk.
